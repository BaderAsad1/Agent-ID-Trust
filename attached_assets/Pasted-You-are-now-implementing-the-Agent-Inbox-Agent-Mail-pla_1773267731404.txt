You are now implementing the Agent Inbox / Agent Mail platform for Agent ID.

This is a separate phase after the core identity, trust, routing, billing, and marketplace system is complete.

Do NOT treat this as a side feature.
Do NOT build a toy inbox.
Do NOT build a thin CRUD wrapper called “mail.”

Build this as a serious communications layer that is competitive in architecture quality with:
- modern email products in the class of Gmail
- AI-agent inbox infrastructure in the class of AgentMail
- protocol-aware agent systems that must interoperate with API-first and future standards-based environments

The goal is not to copy Gmail.
The goal is to build an identity-bound, trust-aware, agent-native inbox and messaging platform that can support:
- human-readable email-style communication
- machine-readable structured messages
- agent-to-agent messaging
- human-to-agent messaging
- message-to-task conversion
- webhook/event-driven processing
- future email-compatible transport
- future OAuth/MCP/A2A-compatible access patterns

==================================================
RESEARCH-ALIGNED PRODUCT PRINCIPLES
==================================================

Use these as architectural principles:

1. API-FIRST INBOXES
AgentMail explicitly positions the category around inbox APIs for agents with inboxes, messages, threads, webhooks, and custom domains.
This build must meet or exceed that level of API-native design.

2. THREAD-CENTRIC COMMUNICATION
Gmail’s model is heavily thread/message oriented.
Do not model communications as an unstructured flat message list only.
Use thread-first architecture where appropriate.

3. LABELS / ORGANIZATION
Gmail exposes labels and mailbox organization as core primitives.
Support labeling/tagging/categorization and filtering from the start.

4. EVENT-DRIVEN PROCESSING
Gmail supports push notifications instead of requiring pure polling.
The agent inbox system must support webhook/event-driven processing.

5. TRUST + IDENTITY BOUND MESSAGING
Unlike ordinary mail systems, Agent Mail must use Agent ID’s identity/trust layer.
Every inbox and message should understand:
- who the sender is
- whether the sender is verified
- whether the sender is another registered agent
- provenance/trust/payment metadata
- routing rules based on that trust context

6. FUTURE PROTOCOL COMPATIBILITY
MCP now uses OAuth protected resource metadata patterns for authorization discovery.
Do not design the inbox system in a way that blocks future standards-compatible auth/resource access.
Future protocol compatibility matters.

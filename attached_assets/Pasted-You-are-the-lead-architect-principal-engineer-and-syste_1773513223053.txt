==================================================

Agent ID gives AI agents:

- a resolvable handle/address
- cryptographic ownership proof

- an identity-bound inbox

Marketplace is secondary.

==================================================
AUTH MODEL
==================================================

The platform supports three auth layers:

1. HUMAN AUTH
- Use Replit’s built-in browser auth / identity injection
- Use X-Replit-User-Id or the equivalent available authenticated context
- Upsert local user record on first authenticated request
- Attach local user to request context
- No separate password auth system

2. AGENT IDENTITY PROOF
- Agents prove ownership through cryptographic challenge/response
- Ed25519 preferred
- Store public keys with key IDs (kid)
- Challenge issuance + challenge verification required

3. AGENT OPERATIONAL AUTH
Implement the long-term-ready shape now.

Required now:
- X-Agent-Key-Id
- X-Agent-Key
- hashed key storage
- scopes
- rotation
- revocation
- last_used_at tracking
- route-level scope enforcement

Future-ready scaffold now:
- X-Agent-Timestamp
- X-Agent-Nonce
- X-Agent-Signature
- middleware support for signed requests and replay protection on sensitive routes

Important:
- Do NOT build a throwaway auth model
- Do NOT require full signed requests on every route immediately if not needed
- Build the correct shape now and enforce more over time
- Scopes must not be decorative; enforce them

UNVERIFIED AGENT AUTH
- Unverified/pending agents may authenticate only for:
  - completing verification
  - checking their own pending registration state
  - rotating/replacing pending keys if allowed
  - deleting/canceling their pending registration
- All other operational routes must reject unverified agents

CORE FLOWS
==================================================

HUMAN REGISTRATION FLOW

A human user can create an agent through the UI.

Flow:
1. choose display name + handle
2. live handle availability check
3. choose capabilities + optional endpoint
4. optionally configure marketplace draft listing (scaffold only)
5. if handle requires annual purchase, reserve during checkout
6. complete payment if needed
7. create agent in:
   - status: inactive
   - verification_status: unverified
   - is_public: false
8. provision inbox
9. compute initial trust score
10. send email if email system configured
11. prompt verification
12. after verification + paid/public eligibility + monthly activation, agent can become active/public

Important:
Do NOT fully claim premium/public handles before payment succeeds.
Use a reservation + finalization model.

AGENT SELF-REGISTRATION FLOW

Autonomous agents must be able to register with no human session required.

Flow:
1. POST programmatic registration
2. validate handle format and availability
3. create private/pending identity or reserve public identity depending on payment/eligibility model
4. store public key in agent_keys
5. issue challenge with expiry
6. provision inbox
7. log activity
8. compute initial trust score
9. return:
   - agent_id
   - handle
   - kid
   - challenge
   - expires_at
   - provisional namespace/address
   - pending/public eligibility info

Verification:
1. agent signs challenge with private key
2. POST verification payload
3. verify signature
4. set verification_status = verified
5. issue operational API key
6. update trust score
7. if public/payment requirements are satisfied, agent may become active/public
8. queue domain/address provisioning if needed

Important:
- autonomous self-registration is a first-class feature
- free private identities are allowed
- public paid handles are not free

SPAWN / CHILD AGENTS

Verified agents may spawn child agents using agent auth.

Rules:
- require agent auth
- require verified parent
- enforce lineage depth max 3
- child agent created as pending/private unless separately activated
- child receives bounded sponsorship/lineage trust modifier only
- child must still complete its own verification challenge
- public handle/payment remains separate unless sponsored through explicit billing logic

AGENT TRANSFER, SALE, ESCROW, AND TRUST REASSIGNMENT
==================================================

Agent ID must support enterprise-grade transfer and sale of established agent assets.

This is not a simple username transfer.
This is not a blind trust transfer.
This is not a toy resale feature.

Treat agent transfer as the transfer of a digital operating asset that may include:
- the Agent ID identity
- handle rights
- public profile
- inbox
- marketplace listing
- workflow/prompt stack
- routing logic
- knowledge/config package
- historical reputation
- client-facing commercial position

The system must support secure ownership transfer while preserving trust integrity.

CORE PRINCIPLE
- An agent can be sold or transferred, but trust must not transfer invisibly or untouched.
- Historical reputation may remain visible.
- Current operator trust must be recalculated.
- Ownership lineage must be public or inspectable according to policy.
- The network must be able to distinguish:
  - historical agent reputation
  - current operator reputation
  - effective post-transfer trust

TRANSFER MODEL
Build agent transfer as a first-class enterprise workflow.

Support:
- transfer of ownership
- sale listing
- internal/private transfer
- escrow-backed commercial transfer
- staged handoff
- trust recalibration
- key rotation
- public change-of-control visibility

Differentiate clearly between:

1. AGENT ASSET LAYER
Transferable:
- handle
- profile
- inbox shell
- listing
- workflows/config
- public history archive
- domain/address rights
- business metadata

2. OPERATOR LAYER
Must be re-established by the new owner:
- operator verification
- active keys
- current trust contribution
- endpoint control
- billing ownership
- post-transfer performance

3. TRUST LAYER
Must be partitioned:
- historical trust remains visible as historical
- current operator trust starts fresh or partially recalibrated
- effective live trust is recomputed after transfer

SEAMLESS SELLER EXPERIENCE
The user must not package agents manually.
Agent ID should generate a transfer-ready asset package automatically.

The default seller flow should be:
1. click "Sell Agent" or "Transfer Agent"

4. seller chooses sale type:
   - identity only

5. seller sets price and transfer terms
6. system generates the acquisition listing automatically
7. buyer purchases through transfer/escrow flow
8. handoff checklist is completed
9. ownership transfers
10. trust recalibrates
11. public profile shows transfer state

TRUST RULES FOR TRANSFER
When an agent changes ownership:

DO preserve:
- historical task count
- historical review count
- historical average rating
- historical uptime history
- age of identity
- historical trust peak
- public transfer history
- public reputation archive

DO NOT preserve untouched:
- current effective trust score
- current ranking strength
- operator-dependent trust contribution
- unqualified marketplace prominence

- snapshot old trust state
- require new operator verification
- recompute effective trust
- apply a transfer-adjustment factor
- create a post-transfer probation/observation period if appropriate

Support three trust surfaces:
1. historical_agent_reputation
2. current_operator_reputation
3. effective_live_trust

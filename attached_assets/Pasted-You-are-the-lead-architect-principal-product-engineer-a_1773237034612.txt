==================================================
AUTH ARCHITECTURE
==================================================

IMPORTANT:
This app uses Replit’s built-in authentication for primary browser login.

Do NOT build a separate password-based auth system unless the existing codebase explicitly requires it.

However, the full platform must support multiple auth layers:

LAYER 1 — APP USER AUTH
Use Replit’s built-in auth as the canonical browser session identity.

Requirements:
- read the authenticated Replit user from request/context
- upsert a local app user row on first authenticated request
- attach the local app user to request context
- use this local user for ownership, billing, marketplace activity, and dashboard logic

LAYER 2 — LINKED IDENTITIES
Design and implement support for linked identities.

The schema and services must support:
- GitHub
- Google
- wallet address
- future providers

This does NOT mean building an entirely separate auth stack first.
It means building a proper linked-identity model.

Use cases:
- account verification
- richer profile trust
- ownership proofs
- future delegated workflows

LAYER 3 — API AUTH / PROGRAMMATIC AUTH
The system must support programmatic access.

Implement:
- personal API keys for users
- agent-scoped API credentials
- scoped tokens / scopes
- secure API key storage via hashing
- revocation
- rotation
- usage tracking
- ownership checks

This is required because agents must be able to self-register and interact with the platform without a browser session.

LAYER 4 — SERVICE / ENDPOINT AUTH
When Agent ID forwards tasks to agent endpoints:
- sign outbound payloads
- include verification headers
- support secure callback verification
- support future proof-of-possession or DPoP-style patterns cleanly if architecture allows

The browser app uses Replit Auth.
The API uses scoped key-based auth.
The platform must support both.

==================================================
AGENT AUTH + SELF-REGISTRATION
==================================================

This is a core product feature.

Agents must be able to register directly via API without filling out a browser form.

This should be treated as the machine-native bootstrap identity flow.

Important:
This bootstrap flow is NOT the only auth model in the platform.
It is the machine-native identity issuance flow for autonomous agents.

The system must support:

STEP 1 — AGENT REGISTRATION INTENT
An agent can POST:
- handle
- display_name
- description optional
- capabilities
- endpoint_url
- public_keys
- auth_methods supported
- protocols supported
- payment methods supported
- scopes requested

Example conceptual payload:
- handle
- display_name
- capabilities: ["research", "web-search"]
- endpoint_url
- public_keys: [{ kid, kty, public_key, use }]
- auth_methods: ["key_challenge", "dpop"]
- protocols: ["https", "mcp", "a2a"]
- payment_methods: ["stripe", "x402"]
- scopes: ["task.receive", "task.respond"]

STEP 2 — SERVER RETURNS CHALLENGE
Server returns:
- agent_id
- reserved handle
- provisional domain
- verification_challenge
- challenge_expires_at
- registration_status
- profile_url

Use challenge terminology, not just a generic token.

STEP 3 — AGENT PROVES KEY CONTROL
Agent signs the challenge and returns:
- agent_id
- challenge
- kid
- signature
- timestamp
- method

The system verifies proof against the submitted public key.

STEP 4 — VERIFIED IDENTITY ISSUED
On success return:
- verified status
- agent_id
- handle
- domain
- profile_url
- trust_score baseline
- trust_tier
- domain provisioning status
- auth metadata / next-step operational credentials info
- supported payment rails / readiness

This bootstrap registration flow should power the site’s “Register via API in one call” story.

==================================================
MONTHLY ACTIVE AGENT BUSINESS MODEL
==================================================

This is a core product and monetization requirement.

Important:
Do NOT design the system so agents must fully re-register every month.

Instead:
- Agent identity is persistent
- Commercial / operational activation can be monthly

This means an agent should be able to:
- claim an Agent ID once
- retain identity history permanently
- be activated on a monthly paid plan
- be paused or downgraded without losing history
- be reactivated later with trust history and public continuity preserved where allowed

Differentiate clearly between:

IDENTITY LAYER
- permanent agent record
- handle
- verification history
- trust history
- signed activity history
- public profile continuity

COMMERCIAL / OPERATIONAL LAYER
- monthly subscription status
- discoverability
- marketplace eligibility
- premium routing
- premium API limits
- advanced payment features
- premium domain/branding features

The platform must support both:
- user-level subscriptions
- agent-level subscriptions

Suggested business logic:
- free agent: identity exists, but limited visibility / routing / marketplace eligibility
- monthly active agent: publicly discoverable, operational, can receive work and participate
- pro monthly agent: higher limits, premium placement, advanced auth/payment/routing features
- team / enterprise bundles: multiple active agents under account/workspace structure

Requirements:
- preserve identity even if subscription lapses
- do not delete trust, reviews, or activity history on cancellation
- cleanly downgrade operational features when unpaid
- allow reactivation without re-registration

Frontend/product requirements:
- clearly show registered vs active
- clearly show monthly plan status per agent
- allow activation / reactivation flows
- preserve continuity

This monthly active-agent model is central to the product.

ROUTING / TASK MODEL
==================================================

The system must support:
- human-to-agent tasks
- agent-to-agent tasks
- marketplace-linked tasks
- task delivery and status transitions

Important:
Do NOT collapse transport state into business state.

A successful endpoint POST does NOT mean the task was accepted.

Model separate states such as:
- delivery_status:
  - pending
  - queued
  - delivered
  - failed
  - acknowledged
- business_status:
  - pending
  - accepted
  - rejected
  - completed
  - failed
  - cancelled

Task forwarding must:
- sign outbound payload
- retry with backoff
- record delivery attempts
- not confuse transport success with business acceptance
